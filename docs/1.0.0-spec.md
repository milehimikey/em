# em 1.0.0 spec: user-defined templates, agent-facing frontmatter, SDD handoff

Status: **proposed** — this is the design reference for the 1.0.0 release. Implementation ships
incrementally as separate PRs against the roadmap in [§11](#11-phased-implementation-roadmap); this
document is the shared spec they build against, the same way [outstanding-items.md](outstanding-items.md)
tracked pre-1.0 work.

## 1. Vision

`em` is **design-phase tooling for Event Modeling, built to be used with AI**. The `.em` DSL and
the `event-modeling` skill exist to get a business process into a rigorous, validated, slice-first
model *before* code is written — the output is a set of implementation-ready slice specs that feed
spec-driven development (SDD).

1.0.0 draws an explicit line: **`em` owns the design phase, not the build phase.**

- **In scope**: modeling the `.em`, writing rich per-slice specs, giving those specs a standard,
  agent-queryable shape (frontmatter), and defining the handoff contract a build-phase tool needs.
- **Out of scope**: task dispatch, enforced build-time status transitions, or any form of agent
  orchestration during implementation. That's the job of external tools — `orchestrate-goal`, a
  future "build-slice" skill, a ticketing integration, whatever a team already uses — which consume
  the standard this document defines.

Today, slice specs are free-form Markdown with a hardcoded template and a prose `**Status:**` line
— fine for a handful of slices, unworkable for the tens-to-hundreds of slices a real model
accumulates. Three problems follow directly from that:

1. Teams want their **own** template sections, not `em`'s opinion of what a slice doc should ask.
2. Nothing about a slice doc is **structured**, so an agent can't filter, version, or search across
   a large slice set without reading every file's dense prose body.
3. Nothing says **what's authoritative** once a slice is "done" — a build tool has to guess which
   part of the doc is the real contract.

1.0.0 fixes all three: user-owned template bodies, a required YAML frontmatter block that's
agent-queryable without reading prose, and a documented SDD handoff contract.

## 2. Frontmatter schema

Every slice doc (`slices/<id>.md`) starts with a YAML frontmatter block. `em` injects and validates
this block regardless of what the user's template body contains (see [§4](#4-user-defined-templates--config)) — it is
never optional, never user-customizable in shape.

### 2.1 Core identity fields (required)

| Field | Type | Notes |
|---|---|---|
| `schemaVersion` | integer | Frontmatter format version. `1` at 1.0.0 launch. Lets a future `em` reject or migrate an unknown format instead of silently misreading it. |
| `id` | string, kebab-case | The **stable key** for this slice, forever. Generated once by `em slice new`, never re-derived from the display name or filename. External tools (tickets, generated code, tests, other slice docs) reference a slice **only** by `id` — titles and filenames are mutable prose, `id` is not. Unique within the model directory (see [§2.4](#24-scoping--assumptions)). |
| `title` | string | Human display name. Mirrors the `.em` slice's name, but the doc is authoritative if they diverge (e.g. renamed in conversation before the `.em` catches up). |
| `pattern` | enum: `state-change \| state-view \| automation \| translation` | The primary **read/write/automation/translation facet**. This is the field an agent filters on to answer "which slices write state" (`state-change`) vs. "which slices are pure reads" (`state-view`) vs. reactions (`automation`, `translation`) — without opening the file body. |
| `status` | enum, see [§3](#3-lifecycle) | |
| `version` | integer ≥ 1 | Bumped on meaningful content change. See [§2.3](#23-versioning). |
| `model` | string | Relative path from this doc to the `.em` file, e.g. `../order-fulfillment.em`. |
| `sliceElement` | string | The `.em` `Element.id` (from `src/model/model.ts`'s `normalize()`) of the element carrying `note "slices/<id>.md"`. Makes the note-link bidirectionally verifiable — `em validate` can confirm the doc and the `.em` element actually point at each other. |

### 2.2 Generated fields — the search/scan layer

These fields exist for one reason: **an agent scanning a model with hundreds of slices must be
able to filter and classify slices from frontmatter alone** — never by re-parsing the `.em`, never
by reading a slice's dense body (field tables, scenarios, invariants). They are **mechanically
generated from the `.em`**, not hand-authored, and are kept fresh by `em slice sync` (see [§6](#6-search--query-cli)),
not by editing YAML by hand.

| Field | Type | Derived from |
|---|---|---|
| `commands` | string[] | Names of `command` elements in this slice. |
| `events` | string[] | Names of `event` elements emitted in this slice. |
| `readModels` | string[] | Names of `view` elements in this slice. |
| `contexts` | string[] | Contexts (`@Context`) touched by this slice's `event` elements. |
| `personas` | string[] | Personas (`@Persona`) touched by this slice's `ui` elements. |
| `triggers` | string (id), optional | For `automation`/`translation` slices: the `id` of the next slice's linked doc, if the command it triggers has one — the "reaction → command" edge of the two-slice split, made queryable without walking the `.em`. |
| `triggeredBy` | string (id), optional | The inverse: set on the command-slice doc, pointing back at the automation/translation slice that triggers it. |

**Why generate instead of derive-at-query-time or hand-author:**

- **Hand-authoring** these would create a second source of truth that silently drifts from the
  `.em` the moment someone edits the model without updating the doc — exactly the failure mode a
  frontmatter standard is supposed to prevent.
- **Deriving at query time** (join doc + `.em` on every `em slice search`) means every search
  re-parses the `.em` and walks the model graph — fine for a handful of slices, a real cost once a
  model has hundreds, and it reintroduces "read the whole system to answer one filter question,"
  which is precisely the problem this schema exists to solve.
- **Generating into frontmatter, kept honest by an explicit sync step and validated for drift** gets
  both: search is a frontmatter-only scan (cheap, no `.em` parse, no body read), and `.em` stays the
  single source of truth because the cache is checkably fresh (`em validate` flags drift; see
  [§5](#5-frontmatter-validation)) rather than silently stale.

### 2.3 Versioning

`version` is a plain integer, bumped by the author (or by `em slice update --bump-version`) when
the doc's *meaning* changes — a new invariant, a changed field contract, a rewritten scenario. It
is **not** semver: a design doc doesn't have a principled definition of "breaking change" the way
an API does, and forcing one would just move the argument into every PR review without adding
signal. **Git log is the actual history and diff mechanism** — `version` is a cheap, human-legible
signal for "has this changed since I last looked," not a replacement for `git log -p`.

A `contentHash` field (sha256 of the body, excluding frontmatter) is stored alongside `version` as
an **internal staleness detector only** — not an identity or a history mechanism. `em validate`
recomputes it and warns if the body changed without a `version` bump. This catches "someone edited
the doc and forgot to bump" — it does **not** enforce that `version` never decreases across commits
(that would require shelling out to `git show HEAD:<path>` on every validate run, which conflicts
with `em`'s zero-system-dependency design for its core commands). Strict monotonic enforcement is
an explicit non-goal for 1.0; revisit only if real usage shows version regressions are a live
problem.

### 2.4 Scoping & assumptions

- `id` uniqueness is scoped **per model directory**, matching the skill's existing one-model-per-
  directory convention. A multi-model monorepo wanting a cross-model registry would need to
  namespace by `model` path + `id` — out of scope for 1.0, flagged here as an assumption rather
  than a silent limitation.
- `.em` element ids (`src/model/model.ts`'s `slug()`) use `_`-separated slugs; slice frontmatter
  `id` and filenames use kebab-case, matching the skill's existing "kebab-case the slice name"
  convention (`SKILL.md`, `slice` phase, step 2). These are two established, distinct conventions
  for two different namespaces (`.em` element identity vs. slice-doc identity) — not a conflict to
  resolve, just don't conflate them.

### 2.5 Optional / authored fields

| Field | Type | Notes |
|---|---|---|
| `swimlane` | `{ persona?: string; context?: string }` | Structured form of the prose "Persona/Actor → Context/Aggregate" line. |
| `upstreamEvents` | string[] | Only for dependencies **not** expressible via `.em` wiring — e.g. an invariant that reads state from an event this slice doesn't formally consume via `from`. If it's derivable from the `.em`, it belongs in [§2.2](#22-generated-fields--the-search-scan-layer), not here. |
| `relatedSlices` | string[] (ids) | Freeform authored cross-references with no structural representation (e.g. "mirrors INV-3 in `place-order`"). |
| `tags` | string[] | Free-text, for `em slice search --tag`. |
| `owner` | string | |
| `created` / `updated` | ISO date | `updated` bumps whenever `version` bumps. |
| `supersededBy` | string (id) | Set by external tools when `status: superseded`. |
| `implementedRef` | string (URL/commit/PR) | Advisory, set by external build tools when `status: implemented`. |

**Design principle**: narrative facts with no `.em` representation stay authored-only. Structural
facts that *do* exist in the `.em` are generated, never hand-duplicated. This is the single rule
that resolves every "should this be a field?" question below — apply it before adding new fields.

### 2.6 Compliance namespace (reserved, unvalidated)

Regulated domains (financial: PCI-DSS, SOX, GAAP; medical: HIPAA; others: GDPR, FedRAMP, ...) need
to track compliance metadata per slice, but their taxonomies differ too much for `em` to validate
one universal schema without either being wrong for most industries or too vague to be useful for
any of them. Rather than adding validated fields, `em` **reserves the top-level `compliance` key**:
it is never written, read for logic, or validated by any `em` command — structurally identical to
how `supersededBy`/`implementedRef` already work as external-tool-owned fields ([§8](#8-sdd-handoff-contract)'s
round-trip-safety guarantee covers this the same way). Teams populate it by hand; `em slice
sync`/`update`/`migrate` shallow-merge and never touch it.

A suggested shape, documented as convention only — adopt what's relevant, ignore the rest, rename
nothing is enforced:

```yaml
compliance:
  frameworks: [PCI-DSS, SOX, GAAP]          # regimes that apply to this slice (any domain's names)
  dataClassification: restricted             # public | internal | confidential | restricted
  sensitiveDataCategories: [PCI, financial]   # freeform: PCI, PHI, PII, financial, biometric, ...
  controlRefs: [PCI-DSS-3.4, SOX-404]         # specific control/citation ids — like INV-n, but external
  retention: 7y                               # retention period, or a link to a retention policy doc
  dataResidency: US-only                      # geographic/legal storage constraint
  reviewedBy: jane@co.com                     # compliance sign-off — distinct from the design `owner`
  reviewedAt: 2026-06-01
```

A medical model uses the identical shape with `frameworks: [HIPAA]`, `sensitiveDataCategories:
[PHI]`, etc. — the *shape* generalizes across industries; the values are each org's own vocabulary.
`em slice search`'s free-text match includes this block ([§6](#6-search--query-cli)) so it stays
findable despite being unvalidated — without that, freeform compliance data would be the one thing
in frontmatter an agent couldn't reliably search for, undercutting the reason it's in frontmatter
at all.

## 3. Lifecycle

```
draft → reviewed → ready-to-implement → in-progress → implemented → (deprecated | superseded)
```

- **`draft` / `reviewed` / `ready-to-implement`** — design-phase states, unchanged names from
  today for continuity. The skill's `slice` phase drives these.
- **`in-progress` / `implemented` / `deprecated` / `superseded`** — post-handoff states.
  **`em` never writes these.** They exist in the enum purely so `em validate`/`em slice *` tolerate
  them when an external build tool (`orchestrate-goal`, a future build-phase skill, a ticketing
  sync) writes them back into the same file. This is decision #1 ([§1](#1-vision)) expressed as a
  schema contract: `em` reads and validates the full lifecycle, but only ever *writes* the
  design-phase states.
- **No enforced ordering.** A rework can legitimately move `implemented` back to `draft`; `em`
  validates enum membership, not transition legality.
- **No in-file status-history log.** Git log is the audit trail per the versioning decision in
  [§2.3](#23-versioning) — this is an intentional simplification, not an oversight. If commit
  discipline around status changes turns out to be unreliable in practice, a `statusHistory` array
  is the natural place to revisit this.

## 4. User-defined templates & config

**`em.config.json`** at the project root:

```json
{
  "sliceTemplate": "templates/my-slice-template.md"
}
```

One field, because it's the one thing 1.0.0 needs: a path to a **body-only** custom template (no
frontmatter in it — `em` always injects that block itself, regardless of what the body contains).
The user's template can define whatever sections they want; `em` only guarantees the frontmatter
prefix, never the body shape.

**Resolution order** (implemented once, in code — not duplicated in the skill's prose):

1. `--config <path>` CLI flag, if given.
2. `em.config.json` in the model directory.
3. Walk up parent directories from the model directory (stop at filesystem root); first match wins.
4. Not found → the shipped default template (the current `templates/slice.md`, becoming body-only
   under this spec — see [§10](#10-skill-updates)).

The skill's `slice` phase stops hand-copying `templates/slice.md` itself and instead calls
`em slice new` ([§6](#6-search--query-cli)), which resolves config and writes the file. This keeps
config resolution in one code path instead of re-implemented (unreliably) in Markdown instructions.

## 5. Frontmatter validation

Extends the existing `Diagnostic { severity: "error" | "warning"; message: string; line?: number }`
pattern in `src/model/validate.ts` with an optional `file?: string`, so slice-doc diagnostics can
report a doc path instead of (or alongside) a `.em` line number.

| Condition | Severity |
|---|---|
| No frontmatter block at all | **error** — distinct message pointing at `em migrate`, not a generic "missing field" |
| Required field missing, wrong type, or invalid enum (`pattern`, `status`) | **error** |
| `id` not kebab-case, or duplicated within the model directory | **error** |
| `version` not a positive integer | **error** |
| `sliceElement` doesn't resolve to a real `Element.id` in the normalized `.em` | **error** |
| `model` path doesn't resolve to an existing `.em` file | **error** |
| `schemaVersion` newer than this `em` build understands | **error** ("upgrade @milehimikey/em") |
| `upstreamEvents` / `relatedSlices` entries that don't resolve | **warning** — soft, these are authored cross-references |
| Orphan doc — no `.em` element's `note` points back at it | **warning** |
| `contentHash` mismatch — body changed without a version bump | **warning** |
| Generated-field drift — recomputing `commands`/`events`/`readModels`/`contexts`/`personas`/`triggers` from the current `.em` doesn't match the stored values | **warning** — "frontmatter out of sync with .em; run `em slice sync`" |

Generated-field drift is **never auto-fixed by `validate` itself** — consistent with `em` never
silently mutating a file outside a command whose entire job is writing that file. Whether this
should eventually change (an implicit sync inside `validate`) is an open call; see [§12](#12-open-judgment-calls).

Wired into `em validate`: after computing `.em` diagnostics, also load and validate every doc in
`slices/` and merge diagnostics before printing. A missing `slices/` directory is not an error (a
model may not have written slice docs yet). New `--skip-slices` flag opts out.

## 6. Search / query CLI

```
em slice new <name> [--dir <model-dir>] [--model <path.em>] [--pattern <p>] [--id <id>] [-f|--force]
em slice sync [id] [--dir <model-dir>] [--all]
em slice list [--dir <model-dir>] [--status <s>] [--pattern <p>] [--format table|json]
em slice show <id> [--dir <model-dir>] [--format table|json] [--body]
em slice search [query] [--dir <model-dir>] [--pattern <p>] [--status <s>] [--context <c>] [--tag <t>] [--format table|json]
em slice update <id> [--dir <model-dir>] [--status <s>] [--bump-version]
```

All subcommands take `--dir`, never a positional directory — `search`'s free-text `query` is
already positional, and a `list [dir]` vs. `search [query] [dir]` split (tried during
implementation) is a real footgun: a bare directory argument silently binds to the wrong
parameter depending on the subcommand. One convention, no ambiguity.

**`list`/`show`/`search` read frontmatter only** — never the body, never a fresh `.em` parse. That
is what makes them cheap at hundreds-of-slices scale:

```
em slice search --pattern automation --context Payment --format json
# → every automation slice touching Payment, answered from frontmatter alone
```

- **`em slice new`** resolves config ([§4](#4-user-defined-templates--config)), writes
  `slices/<id>.md` with the injected frontmatter block (`id` via a shared `slug()` util,
  `schemaVersion: 1`, `status: draft`, `version: 1`, `created`/`updated` = today) plus the
  configured or default body, then runs an initial `sync`. It does **not** touch the `.em` — `em`
  never mutates hand-authored `.em` source outside `init` writing a brand-new file. Wiring
  `note "slices/<id>.md"` into the `.em` stays the agent's job (unchanged from today's `slice`
  phase step 3).
- **`em slice sync [id | --all]`** is the *only* command that reads the `.em` to (re)compute the
  generated fields ([§2.2](#22-generated-fields--the-search-scan-layer)) and shallow-merges them
  back into the doc's frontmatter — hand-authored and external-tool fields are preserved, never
  overwritten wholesale. Run automatically by `em slice new` at creation, and manually (or
  agent-invoked, per the skill's updated `slice` phase — [§10](#10-skill-updates)) after `.em`
  edits that change a slice's elements.
- **`em slice update`** is a proposed *addition* beyond the literal ask — a single code path for
  hand-authored writes (`status`, `version` bump, tags, etc.), so external-tool-owned fields
  (`supersededBy`, `implementedRef`, arbitrary custom fields a build tool adds) survive instead of
  being clobbered by an agent hand-editing YAML. Flagged explicitly as new, not requested outright.

`dir` defaults to cwd; commands expect exactly one `*.em` file in it (today's skill convention)
unless `--model` disambiguates.

**No cached index file.** Design-phase scale is tens to low-hundreds of docs per model — parsing
that many small YAML frontmatter blocks is single-digit milliseconds; a cache would introduce the
first staleness/invalidation problem in an otherwise stateless-reads tool (mtime tracking, a
`reindex` step people forget to run). If a project ever grows past that, an opt-in `--cache` is a
reasonable future addition — explicitly out of scope for 1.0.

## 7. `em migrate`

```
em migrate <model-dir> [--dry-run] [--force] [--report <path>]
```

One-time upgrade from the pre-1.0 fixed-template format to the frontmatter standard. Refuses a
dirty git working tree unless `--force` — git is the undo mechanism for a rewrite tool, so it only
makes sense to run against a clean tree. `--dry-run` prints the plan without writing. Idempotent: a
doc that already has `schemaVersion` is skipped as "already migrated."

Per `slices/<name>.md`:

1. **Prefer inferring from the `.em` over parsing prose.** Load and normalize the `.em`, find which
   element's `note` points at this doc, derive `pattern` from that element's kind (`command` →
   `state-change`, view-only → `state-view`, `processor`/`automation`/`saga` → `automation`,
   `translation` → `translation`) and `sliceElement` from its `Element.id`. This is more reliable
   than parsing the old template's freeform `{{State Change | ...}}` line, which is often left
   unfilled or inconsistent.
2. Fall back to regex-parsing the old `**Pattern:**`/`**Status:**` prose bullets only for orphan
   docs with no `.em` link. Unmatched status → default to `draft` **and** flag it loudly in the
   migration report — never guess upward (a wrong "ready-to-implement" is worse than an honest
   "draft").
3. `id`: slugify the doc's `# Slice: {{Name}}` heading or the linked `.em` slice's name, via a
   `slug()` helper shared with `.em` element-id generation for consistent collision handling
   (`-2`, `-3`, … suffixes).
4. `version: 1` for everything — no prior version concept existed; this is an explicit
   simplification consistent with git being the real history per [§2.3](#23-versioning).
5. `created`/`updated`: both set to the migration run date, not reconstructed from file
   `birthtime` or git-log archaeology (unreliable) — an intentional lossy simplification.
6. `upstreamEvents`: best-effort extraction from the old "Dependencies & Read Models Affected"
   bullets; left empty and flagged if unparseable.
7. Run an initial `sync` per doc so generated fields ([§2.2](#22-generated-fields--the-search-scan-layer))
   aren't just defaulted to empty.
8. **Body cleanup**: strip the now-frontmatter-only `**Pattern:**`/`**Status:**` bullet lines
   (they'd otherwise duplicate frontmatter and drift); leave everything else untouched.

Also rewrites `.event-modeling.md`: adds a small `schemaVersion`/`model` frontmatter block, and
regenerates the "Slice inventory" table's status column from the migrated docs' frontmatter
(id + pattern + status + version), replacing the old free-text "none/draft/ready".

Emits a migration report (stdout, or `--report <path>.md`) listing every file touched and every
field defaulted or left unmatched rather than confidently inferred.

## 8. SDD handoff contract

This is the interface a build-phase / SDD tool is expected to consume. `em` documents it; it does
not implement a consumer (per the design-phase-only scope in [§1](#1-vision)).

**"Ready" is defined as**: `status` is `ready-to-implement` or later, **and** `em validate` reports
zero errors for both the `.em` model and this doc's frontmatter.

**Authoritative for a build tool:**

- **`id`** — the only stable key for linking tickets, PRs, generated code, and tests back to the
  slice. Never key on title or filename, both of which are mutable prose.
- **The Command/Input field table** — the parameter/type/required/rules contract for the handler
  signature. This is richer than the `.em`'s lightweight inline `{ field: Type }` blocks, which
  stay a structural preview only. **Pre-existing, documented gap**: `docs/features.md` already
  notes information-completeness validation (tracing `.em` inline fields to slice-doc fields) is
  planned but not shipped in 1.0 — the `.em` and the slice doc's field table are not cross-
  validated, so they can diverge. **The slice doc wins** when they do.
- **The Event(s) Emitted table**, with "Immutable Fact? yes" columns treated as compile-time-
  enforced immutability by generators.
- **`INV-n` invariant ids** — traceable rule references. Generated tests should cite them by id
  (e.g. `test("INV-1: rejects order over credit limit")`) — recommended, not enforced by `em`.
- **Given/When/Then scenarios** — acceptance criteria. Happy path is the minimum viable test;
  rule-boundary scenarios trace to their `INV-n`.
- **Alternate & Error Flows** — required test cases (retries, idempotency, compensation), not
  optional extras.
- **Derived relationships** (`triggers`/`triggeredBy`, plus upstream/downstream computed by joining
  doc + `.em`) — a dependency graph a build orchestrator can use for sequencing (don't build a
  slice until its upstream slices are at least `implemented`). Advisory only; `em` doesn't enforce
  sequencing.
- **`compliance.reviewedBy`/`controlRefs`** ([§2.6](#26-compliance-namespace-reserved-unvalidated)),
  if present — a build tool *may* treat these as a gating signal (e.g. refuse to mark a slice
  `implemented` without a compliance sign-off). Advisory only, same as `triggers`/`triggeredBy`;
  `em` has no opinion on whether a given slice needs one.
- **Round-trip safety guarantee**: any `em`-side write to frontmatter (`em slice update`, `em slice
  sync`, a re-run of `em migrate`) shallow-merges, never wholesale-regenerates — external-tool-
  owned fields (`supersededBy`, `implementedRef`, any custom field a build tool adds) must survive.
- **Statuses from `in-progress` onward are out of `em`'s write scope** — it must tolerate them,
  never write them.

## 9. File / module layout

New, under `src/`:

| File | Purpose |
|---|---|
| `src/config.ts` | Config type + resolution ([§4](#4-user-defined-templates--config)). |
| `src/model/frontmatter.ts` | YAML frontmatter split/parse/stringify, backed by the `yaml` package (comment/field-preserving round-trip — see dependency note below). |
| `src/model/sliceDoc.ts` | `SliceFrontmatter` type ([§2](#2-frontmatter-schema)), `SliceDoc { path, frontmatter, body, raw }`, `loadSliceDocs(modelDir)`. |
| `src/model/validateSliceDocs.ts` | The diagnostics in [§5](#5-frontmatter-validation). |
| `src/slice/generate.ts` | `em slice new` logic. |
| `src/slice/sync.ts` | Derives `commands`/`events`/`readModels`/`contexts`/`personas`/`triggers` by walking the normalized `.em` model from `sliceElement`, reusing `NormalizedModel.byId`/`elements`/`slices` already built by `src/model/model.ts`. |
| `src/slice/query.ts` | `list`/`show`/`search`/`update` logic (pure functions; CLI wires flags + printing). |
| `src/migrate/migrate.ts` | `planMigration()` / `applyMigration()` (kept separate so `--dry-run` is just "don't call apply"). |
| `src/util/slug.ts` | Extracted from `src/model/model.ts`'s private `slug()`, shared by `model.ts`, `slice/generate.ts`, and `migrate/migrate.ts`. |

Modified:

- `src/cli.ts` — new `em slice` command group, `em migrate`, extend `validate` to load and merge
  slice-doc diagnostics.
- `src/model/validate.ts` — add optional `file?: string` to `Diagnostic`, update `formatDiagnostic`.
- `package.json` — add `yaml` dependency.

**New dependency**: `yaml` (eemeli/yaml, pure JS, no native bindings). Needed because `em slice
update`/`sync`/`migrate` write back into these files repeatedly, and `yaml`'s `Document` API
preserves comments/formatting/unknown-field ordering on incremental edits — a hand-rolled parser
would need real work to avoid clobbering fields written by external tools, which is a hard
requirement per [§8](#8-sdd-handoff-contract)'s round-trip guarantee. Flagged explicitly: this is
dependency #5 in a project that currently has 4 and markets self-containment — but it's pure JS,
unlike the WASM/native `@hpcc-js/wasm-graphviz`/`@resvg/resvg-js` deps, so the risk profile is low.

## 10. Skill updates

- `templates/slice.md` becomes **body-only** — no frontmatter (that's injected by `em slice new`);
  drop the now-redundant `**Pattern:**`/`**Status:**` prose bullets.
- `templates/state.md` gets a small frontmatter block and an `Id` column added to the Slice
  inventory table.
- `SKILL.md`'s `slice` phase: doc creation goes through `em slice new` instead of hand-copying the
  template; add a step — "after any `.em` edit affecting this slice's elements, run

## 11. Phased implementation roadmap

Status legend: 🔴 not started · 🟡 in progress · 🟢 done (matches `outstanding-items.md`).

- **Phase A — Frontmatter core** 🟢 (done): `src/model/frontmatter.ts` (parse/stringify/merge a
  `---` YAML block via the `yaml` package's `Document` API, preserving unknown fields on write),
  `src/model/sliceDoc.ts` (`SliceFrontmatter`/`SliceStatus`/`SlicePattern` types, `SliceDoc`,
  `loadSliceDocs()`), `src/util/slug.ts` (extracted `slug()` + new `kebabSlug()`/`dedupe()`,
  `model.ts`'s `normalize()` now imports from here instead of a private copy). No CLI surface yet
  — that's Phases C/D.
- **Phase B — Validation** 🟢 (done): `src/model/deriveSliceFields.ts` (shared generated-field
  derivation, reused by Phase C's `sync`), `src/model/validateSliceDocs.ts` (all checks in §5),
  wired into `em validate` (new `--skip-slices` flag), `Diagnostic` extended with `file?: string`.
  A model directory with no `slices/` still validates exactly as before (no-op).
- **Phase C — Config + `em slice new`/`sync`** 🟢 (done): `src/config.ts` (`em.config.json`
  resolution, walks up to filesystem root), `src/slice/generate.ts` (`em slice new`, dedupes ids
  via shared `kebabSlug`/`dedupe`), `src/slice/sync.ts` (`em slice sync`/`syncAll`). **Addition
  beyond the original spec text**: `sync` auto-discovers `sliceElement` from the `.em`'s `note`
  back-reference when a doc doesn't have one set yet, instead of requiring it up front — this
  closes a real chicken-and-egg gap (`em slice new` runs before the agent has wired the `note`
  into the `.em`, so `sliceElement` can't be known at creation time). `src/util/findModel.ts`
  added (shared "find the one `.em` in this dir" resolution, not in the original file table).
- **Phase D — Query CLI** 🟢 (done): `src/slice/query.ts` (`list`/`show`/`search`/`update`).
  **Deviation from the original spec text**: all `em slice *` subcommands use `--dir <path>`
  uniformly, not a positional `[dir]` — a `list [dir]` vs. `search [query] [dir]` split was
  implemented first and caught during smoke-testing: a bare directory argument silently bound to
  `query` instead of `dir`. One convention avoids the ambiguity.
- **Phase E — `em migrate`** 🟢 (done): `src/migrate/migrate.ts` (`planMigration`/`applyMigration`/
  `formatMigrationReport`/`assertCleanWorkingTree`), including Slice inventory table regeneration
  in `.event-modeling.md` — rows are matched to migrated docs by title (or by kebab-slugging the
  row's slice name) and rewritten with an added `Id` column; unmatched rows (e.g. an unfilled
  `{{Slice Name}}` template row) are left untouched and flagged in the migration report rather
  than guessed.
- **Phase F — Skill rewrite** 🟢 (done): `templates/slice.md` (body-only now, `**Pattern:**`/
  `**Status:**` bullets dropped, `**Swimlane:**` kept per §12.1); `templates/state.md` (frontmatter
  block + `Id` column); `templates/model-readme.md` (Status column, optional per §10, included);
  new `reference/frontmatter.md` (schema/lifecycle/handoff quick reference + CLI); `em-dsl.md`
  (CLI reference section); `SKILL.md` (reading list, pre-1.0 migration check in Preconditions, the
  `slice` phase rewritten around `em slice new/sync/update`, `validate` phase covers slice-doc
  diagnostics, command reference updated).
- **Phase G — 1.0.0 release** 🟡 (in progress): `package.json` and the CLI's `--version` bumped to
  `1.0.0` (also fixed a pre-existing drift — the CLI had `.version("0.2.0")` hardcoded against
  `package.json`'s `0.2.1`); README updated (Commands, a new "Slice docs & frontmatter" section,
  Documentation links); `docs/features.md` updated. **`npm publish` intentionally not run** — that
  makes this version world-visible and irreversible in a way that warrants an explicit,
  separate go-ahead rather than happening as a side effect of finishing the roadmap.
- **Phase H — Compliance namespace convention** 🟢 (done): [§2.6](#26-compliance-namespace-reserved-unvalidated)
  (reserved, unvalidated `compliance:` block — schema stays domain-agnostic, teams document their
  own regulatory metadata under one collision-free key); `em slice search`'s free-text match
  extended to include it (`src/slice/query.ts`) so it stays findable despite being unvalidated;
  `reference/frontmatter.md`, `templates/state.md` (Regulatory scope session input), and
  `SKILL.md` (ask about it once per model, then per-slice only when relevant) updated to match.

Each phase is PR-sized and independently reviewable, consistent with how Notes/Fields/Public-prep
shipped pre-1.0.

## 12. Open judgment calls

Flagged here rather than resolved unilaterally — worth a decision before or during implementation:

1. **`**Swimlane:**` prose** — drop it from the slice template body alongside `**Pattern:**`
   (redundant with frontmatter's `swimlane` field), or keep it since it reads more like narrative
   than structured metadata? Recommendation: keep as prose; it's context for a human reader in a
   way `swimlane: {persona, context}` alone isn't.
2. **Build-tool bookkeeping fields** (e.g. `assignee`, `claimedBy`) — add now, or stay strictly
   hands-off per the design-phase-only scope? Recommendation: leave out of 1.0; it's a one-line
   schema addition later if a build tool needs it, and adding it speculatively risks scope creep
   into build-tool territory this spec deliberately avoids.
3. **Monotonic version enforcement** — 1.0 only catches same-commit drift (`contentHash` vs.
   `version`), not cross-commit regression (would need a `git show` shell-out). Revisit only if
   real usage shows version regressions happening in practice.
4. **Generated-field drift: warn-only forever, or auto-fix via implicit sync inside `validate`?**
   Recommendation: warn-only for 1.0, consistent with `em` never silently mutating a file outside
   the command whose job is writing it — but this is worth revisiting once real usage shows how
   often docs actually drift out of sync.
